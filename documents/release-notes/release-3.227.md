# Flow Tool Kit v3.227 Release Notes

## New Features

### Inline Label Width: Admin-Configurable Column Width

Form fields configured with **Label Variant = Label Inline** now expose a new **Inline Label Width** slider in the form field configuration. Admins can tune the label column width per field, from 10% to 90%, with a default of 33% that matches Salesforce's standard SLDS inline-label layout.

**Why.** Long inline labels previously squished into a narrow column or pushed prepend addons (like `$` or `https://`) onto their own line. Stacked inline-label fields with mixed-length labels never aligned consistently. With the new slider, admins control the column width, long labels wrap cleanly inside the configured column, and stacked fields align on a shared vertical edge.

**How it works.** The slider stamps a CSS percentage value on the field record. At runtime, `flowFormField` injects that value into a `--flowtoolkit-inline-label-width` CSS custom property on the field wrapper, which propagates through the shadow DOM into Salesforce's internal `slds-form-element_horizontal` rules, overriding the SLDS default `calc(33% - 1.25rem)` cap that previously caused the squish.

![Inline Label Width slider demo](https://raw.githubusercontent.com/common-unite/Flow_Tool_Kit_Public/main/documents/screenshots/106-inline-label-width-demo.gif)

**Affected components:**
- `Form_Component_Field__c.inlineLabelWidth__c`: new Text(20) field on the form field SObject
- `formBuilderSectionField` LWC: adds the slider input, gated on `labelVariant === 'label-inline'`
- `flowFormField` LWC: `fieldStyle` getter applies the CSS custom property at render time
- `formOverrideStyles.css`: overrides SLDS `slds-form-element_horizontal` label / control rules to honor the variable
- `Form_Builder_Admin`, `Form_Builder_Manager`, `Form_Flow_User` permission sets: granted FLS for the new field

## Bug Fixes

### Picklist Values with Long Labels Stay Inside the Field Column

Combobox and picklist fields with long option labels (e.g. *"Annual Revenue Generated By Last Year's Top Performer Award Winners"*) were extending past the form column, growing the form section excessively wide and breaking column / section sizing.

**Root cause.** The `lightning-layout-item` wrapping each combobox is a flex child of the field's outer `slds-grid` container. With `slds-grow` (`flex: 1 1 auto`) and a default `flex-basis: auto`, the column was sized to the combobox's intrinsic content width (the longest option label) and refused to shrink.

**Fix.** A scoped CSS override forces the combobox layout-item to `flex: 1 1 0` with `min-width: 0`, so the column takes only the available space (after any prepend or append siblings). The platform's built-in `slds-truncate` class on the combobox button then engages correctly, truncating long option labels with ellipsis instead of overflowing.

### Calculate Record Output Renders Immediately in Repeater Headers (Experience Cloud)

The Repeater component's calculated totals (Sum, Min, Max, Average aggregations) now render immediately in built-in headers and in text-template / formula references that consume `RepeaterScreen.calculatedResult` on Experience Cloud sites (both Aura and LWR). Previously the header rendered empty on initial load and only populated after a user interaction with the form.

**Root cause.** A timing race between form metadata loading and records arrival. The Repeater's calculation runs inside `resetRecords` → `dispatchCalculatedData` → `calculatedFields`. The `calculatedFields` getter sets a sticky `skipCalculationsNoCalculatedFields = true` flag when no calculation columns are found, intended for forms that have no aggregations configured. In Experience Cloud, the Apex `Form_Controller.getForm` call delivers form metadata **later** than in Lightning record-page Flow runtime, late enough that records arrive first, the calculation runs against an empty field set, the sticky flag flips on, and even after form metadata finally arrives the calculation is permanently disabled.

**Fix.** When form metadata arrives in `handleFormMetaDataResult`, detect that the sticky flag was set (which uniquely indicates the calc previously attempted and bailed because the form wasn't loaded), clear the flag, and re-trigger `dispatchCalculatedData`. The flag itself is the signal; the retry only happens when retry is needed.

### Sensitive (Encrypted) Fields Route Through Salesforce's UI API

Classic Encrypted (`EncryptedString`) and Shield Platform Encryption fields now render through the same `lightning-input-field` UI-API path used by reference (lookup) fields, instead of a plain `lightning-input` of type text.

**Why.** Per Salesforce documentation, `lightning-input-field` natively handles encrypted text fields with the platform's intended masking behavior (clear text during typing, masked after save) and respects the running user's *View Encrypted Data* permission when reading saved values via `lightning/uiRecordApi`. Routing encrypted fields through this path aligns Form Builder with Salesforce's standard record-page rendering for sensitive data.

**Affected components:**
- `flowFormField.js`: new `isEncrypted` getter (covers `dataType === 'encryptedstring'` and Shield's `objectInfoField.encrypted === true`); `isInputField` extended to route encrypted fields through the UI-API slot path
